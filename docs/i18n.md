# Internationalization (i18n) Dictionary

key files: `lib/i18n.ts`, `lib/resolve.ts`, `lib/tokens.ts`, `lib/experiment-validation/check-dictionary-references.ts`

An experiment can declare a **dictionary** of localized messages and address them
from any user-visible string prop with the `[[key]]` token. This keeps
participant-facing copy out of the graph structure and lets one experiment serve
multiple languages from a single config.

## Declaring a dictionary

Add two optional fields to `ExperimentFlow`:

```ts
const experiment: ExperimentFlow = {
  defaultLocale: 'es',
  dictionary: {
    es: { greeting: 'Hola', cta: { start: 'Empezar' } },
    en: { greeting: 'Hello', cta: { start: 'Start' } },
  },
  nodes: [...],
  edges: [...],
  screens: [...],
};
```

- `dictionary: Record<locale, MessageTree>` ã one message tree per locale. Trees
  may nest arbitrarily; a nested branch like `cta: { start: 'ãÎ' }` is addressed by
  the dotted key `[[cta.start]]`. Flat dotted keys (`'cta.start': 'ãÎ'`) are also
  allowed and may be mixed with nesting ã both flatten to the same dotted key.
- `defaultLocale?: string` ã the locale used when no valid `?lang=` is provided,
  and the source of fallback messages for keys missing in the active locale. When
  omitted, the first declared locale acts as the default.

Keys may contain word characters, dots, and hyphens (`welcome.title`,
`cta-start`) ã the same charset as `{{ }}` data references. At runtime each
locale's tree is flattened to dotted keys (`buildMessages` ã
`flattenMessages`), so `context.messages` stays a flat `Record<string, string>`.

## Referencing messages: the `[[ ]]` token

Use `[[key]]` anywhere `{{ }}` answer-piping is supported (labels, option labels,
`content`, `placeholder`, button `text`, slider min/max labels, image `alt`/`url`):

```ts
{ componentFamily: 'layout', template: 'button', props: { text: '[[cta.start]]' } }
```

`[[ ]]` is distinct from `{{ }}`:

| Token | Resolves against | Purpose |
|---|---|---|
| `[[key]]` | the active locale's dictionary (`context.messages`) | localized static copy |
| `{{$$path}}` etc. | runtime context data | answer piping |

## Resolution order and nesting

`resolveValuesInString` runs the dictionary pass **first**, then the `{{ }}` pass.
This means a dictionary message may itself contain answer-piping or nested
references, and they all resolve:

```ts
dictionary: {
  en: {
    'thanks.body': 'All done, {{$$welcome.name}}!', // {{ }} inside a message
    intro: 'Welcome ã [[greeting]]',                 // nested [[ ]]
  },
}
```

Both passes share a depth guard (max 10), so self-referential messages do not
overflow.

**No re-entry across passes.** The `[[ ]]` pass runs to completion before the
`{{ }}` pass begins. A `[[key]]` that only appears *after* `{{ }}` resolution
(e.g. a participant's piped answer that happens to contain `[[other]]`) is **not**
re-interpreted as a dictionary token ã it renders literally. This makes
participant-supplied data safe: it can never forge a dictionary lookup.

> The `dictionary-key-collision` validator guards the one ambiguous case: within
> a single locale, a flat `"a.b"` key and a nested `a: { b }` both flatten to the
> dotted key `a.b`. Mixing nested and flat keys is fine as long as they don't
> collide on the same dotted path.

## Dynamic dictionary keys

A `[[ ]]` key may itself contain `{{ }}` answer-piping tokens, so the key can be
computed from runtime data:

```ts
content: '[[experience.{{$$screen.drug}}]]'
// with $$screen.drug === 'lsd' ã renders the experience.lsd message
```

The inner `{{ }}` are resolved **during the dictionary pass, before the lookup**:
the key reduces to a plain dotted key (`experience.lsd`), which is then looked up
in the active locale. A resolved message may still contain its own `{{ }}` tokens,
resolved by the subsequent answer-piping pass as usual.

**Misses render the reduced literal.** If `experience.lsd` is undefined, the token
renders `[[experience.lsd]]` (the reduced key), telling you exactly which key was
missing. If the inner `{{ }}` itself can't resolve, the token renders unreduced,
e.g. `[[experience.{{$$screen.drug}}]]`. Misses are never silently blank ã matching
the `{{ }}` philosophy.

**The no-re-entry guarantee still holds.** Only `{{ }}` written *lexically inside*
`[[ ]]` is resolved into a key. A `[[other]]` that appears *after* `{{ }}` resolution
ã e.g. inside a participant's piped answer ã is still never re-interpreted as a
dictionary token, so participant data can never forge a dictionary lookup. **Feed
dynamic keys from constrained sources (fixed option sets, compute outputs), not free
text.**

**Validation is family-level.** The validator checks that the static stem of a
dynamic key (e.g. `experience.`) matches at least one defined key; it cannot know the
concrete runtime value, so it can't verify the *specific* key will exist. Per-key
parity across locales is still covered by `dictionary-locale-mismatch` over all
defined keys.

The validator assumes each `{{ }}` fills **exactly one** path segment (no dot) ã the
common case where a value comes from a fixed option set or compute output. This is
stricter than the runtime, which would look up a dot-containing value spanning
segments. So if a `{{ }}` value can resolve to a string containing a dot (e.g.
`"lsd.variant"`), the validator reports `unknown-dictionary-key` even though the
runtime could resolve it. Keep dynamic-key values single-segment to avoid that
false positive.

## Selecting the active locale

The locale is chosen from the `?lang=` query parameter, mirroring how
`?condition=` selects a start node:

```
/experiments/i18n-demo?lang=en   ã English
/experiments/i18n-demo           ã defaultLocale
/experiments/i18n-demo?lang=zz   ã unknown ã defaultLocale
```

`selectLocale(searchParams, experiment)` (`lib/i18n.ts`) returns the active
locale; `buildMessages(experiment, locale)` flattens that locale's messages with
the default locale merged underneath as fallback. `startExperiment(..., locale)`
seeds `context.locale` and `context.messages`, which then travel with the
context through the whole run.

## Missing keys and fallback

1. Active locale has the key ã use it.
2. Otherwise the **default locale's** value is used (merged in by `buildMessages`).
3. Otherwise the token is left **literal** (`[[key]]`), so typos are visible in
   development ã matching the `{{ }}` philosophy.

## Validation

`validateExperiment` runs `checkDictionaryReferences`. It scans `[[ ]]`
references in screen component props **and** shared-option labels
(`ExperimentFlow.options`) ã everywhere `resolveValuesInString` runs ã emitting:

- `unknown-dictionary-key` ã a `[[key]]` is referenced but defined in no locale.
- `dictionary-locale-mismatch` ã a key exists in some locales but not others
  (warning).
- `unknown-default-locale` ã `defaultLocale` is not a key of `dictionary`.
- `dictionary-key-collision` ã within one locale, a flat dotted key and a nested
  path flatten to the same dotted key.
See `docs/validate.md` for the full error-code list.

## Worked example

`src/data/experiments/i18n-demo.ts` (registered as `i18n-demo`) is a complete
bilingual (es/en) experiment with nested message trees: every string is a
`[[ ]]` reference, the survey
question and thank-you message pipe the participant's name via `{{ }}` inside a
localized message, and `?lang=` switches the whole experience.
