# Screen Form Validation

Documents how `lib/validation.ts` decides whether a screen's data is complete and valid before advancing to the next step. Covers what gets checked, when checks run, how complex layouts (conditionals, for-each loops) are handled, and what the current limitations are.

---

## What this module does

Every screen that collects responses needs a contract: which fields are required, what format they must be in, and — for fields that only appear under certain conditions — whether they are present at all at submit time. This module builds that contract from the screen definition and enforces it when the participant clicks the submit button.

It does not render anything. It only produces a validation function that React Hook Form calls on submit.

---

## The two validation phases

Validation happens in two passes that run sequentially on submit.

### Phase 1 — Static schema (built at screen load)

When the screen is first rendered, the module reads the screen definition and builds a fixed set of rules — one per field — from the component tree. This is done once and does not change while the screen is visible.

Each field's rules come directly from its component props: whether it is required, minimum/maximum length, allowed number range, regex pattern, etc.

Fields that are **always present** (top-level responses, inside groups, inside static for-each loops) are added as required or optional depending on their `required` prop.

Fields that are **conditionally present** (inside a `conditional` component) are always registered as optional in the static schema. This is intentional: if the condition is false and the field never appears, the schema should not reject the data for missing it. Whether it is actually required is enforced in Phase 2.

Fields generated by a **dynamic for-each** loop are not registered in the static schema at all, because their names (e.g. `eats-apple`, `eats-banana`) are not known until the participant fills in the source field that drives the loop. They are handled entirely in Phase 2.

### Phase 2 — Runtime checks (run on every submit attempt)

After the static schema passes, a second pass walks the full component tree recursively against the live form data. It carries two pieces of state as it descends:

- **For-each context** — a map of loop id to `{value, index}` that accumulates as the walk enters for-each loops, so that `{{#id.value}}` placeholders resolve correctly at any nesting depth.
- **Dynamic mode** — a flag that starts `false` and flips to `true` the first time the walk enters a conditional branch or a dynamic for-each iteration. Fields are only checked in dynamic mode; everything else is already covered by Phase 1.

**Conditionals:** when the walk reaches a conditional, it evaluates the condition using the live form data (including any accumulated for-each context). If true, it descends into the `if` branch in dynamic mode. If false and an `else` branch exists, it descends into that instead. Fields in whichever branch is not taken are never validated.

**Dynamic for-each:** the walk reads the source array from the live form data, then iterates over it. For each value it updates the for-each context and descends into the template in dynamic mode.

**Nesting:** because the walk is recursive and carries context, any combination of conditionals and for-each loops — at any depth — is handled correctly. A conditional inside a for-each resolves its condition using the surrounding loop's value. A nested conditional is only reached if the outer conditional's branch was entered.

If either phase fails, the errors are collected and shown to the participant. The form does not advance.

## Component types and what gets validated

| Component | Expected value | Required check | Other constraints |
|---|---|---|---|
| `text-input`, `text-area` | Non-empty string | `min(1)` | `minLength`, `maxLength`, `pattern` |
| `date-input`, `time-input` | Non-empty string | `min(1)` | — |
| `dropdown`, `radio` | Non-empty string | `min(1)` | — |
| `likert-scale` | Non-empty string | `min(1)` | — |
| `checkboxes` | Array of strings | At least 1 selected | `min` (minimum count), `max` (maximum count) |
| `numeric-input` | Number | Present and valid | `min`, `max` |
| `slider` | Number or null | Must not be null/undefined | `min`, `max`, `minValue`, `maxValue` |
| `single-checkbox` | Boolean | Must be `true` | `shouldBe` (can require `false` instead) |

### Required vs optional

Every component's `required` prop defaults to `true` if omitted. Setting `required: false` makes the field optional — it may be submitted empty without error.

### Custom error messages

Every component accepts an `errorMessage` prop. When set, it replaces the default "This field is required" message for that component's required check. Length and range constraints also support their own `errorMessage` per constraint.

### Slider: a special case

A slider's value starts as `null` (not touched) and becomes a number once the participant moves the handle. `required: true` on a slider means the participant must have interacted with it — `null` is rejected. `required: false` accepts `null` but still validates the range if a value is present.

### Single checkbox with `shouldBe`

`shouldBe: true` requires the checkbox to be checked. `shouldBe: false` requires it to be unchecked. This is used for things like "you must have read the instructions" (checked) or "you must not opt out" (unchecked). When `shouldBe` is set it takes precedence over `required`.

## Groups

A `group` component is a layout container with no validation logic of its own. Its child components are validated exactly as if they were at the top level. Groups exist for visual organisation and have no effect on whether fields are required or optional.

---

## Static for-each

A static for-each loop has a fixed list of values defined in the screen config. The module expands it at screen load time: if the loop has three values (`apple`, `banana`, `kiwi`) and the template field's key is `likes-{{#fruits.value}}`, three fields are added to the static schema: `likes-apple`, `likes-banana`, `likes-kiwi`. Each one is validated independently with the same rules as the template.

Template keys support two substitutions:

| Placeholder | Replaced with |
|---|---|
| `{{#id.value}}` | The string value at that iteration (e.g. `apple`) |
| `{{#id.index}}` | The zero-based position (e.g. `0`, `1`, `2`) |

Both can be combined in the same key: `item-{{#id.index}}-{{#id.value}}` → `item-0-apple`.

---

## Dynamic for-each

A dynamic for-each loop's values come from another field collected in the same screen (or from experiment-wide data). Because those values are only known at runtime, the resulting field names cannot be computed at screen load.

**What happens at load time:** nothing. These fields produce no static schema entries.

**What happens at submit time:** the Phase 2 walk reads the source array from the submitted form data, iterates over it, substitutes `{{#id.value}}` and `{{#id.index}}` into the template key, and checks each resolved field's required rule. If the source array is absent or empty, no checks run.

Example: source is `$countries-known` = `['france', 'italy']`, template key is `traveled-{{#fe.value}}`. On submit, the module checks `traveled-france` and `traveled-italy`.

### Nesting inside other structures

A dynamic for-each can be placed inside a conditional (it is only validated when that branch is active), inside a static for-each (the outer loop's context is available for key resolution), or both. All combinations are handled correctly.

### The passthrough mechanism

Because dynamic for-each fields are not in the static schema, Zod would normally strip them from the submitted data before it reaches the server. To prevent this, the schema is built with `passthrough()`, which tells Zod to keep any keys it does not recognise. This is the reason dynamic for-each fields reach the server intact.

---

## Conditionals

A conditional component shows one subtree when its condition is true and optionally a different subtree (the `else` branch) when it is false. The module handles this in two steps:

1. All fields inside any conditional branch are registered as optional in the static schema (Phase 1 does not reject the form if they are absent).
2. On submit (Phase 2), the condition is evaluated against the live form data. If true, the fields inside the `if` branch are checked for required values. If false and an `else` branch exists, its fields are checked instead. Only the active branch is validated.

Conditionals can be nested. The inner conditional is only reached — and its condition only evaluated — if the outer conditional's branch was entered. A required field three levels deep is only validated when all three enclosing conditions are met.

---

## Template key syntax in conditions

A condition's `dataKey` can itself contain a `{{#id.value}}` placeholder. This is used when a conditional lives inside a for-each loop and the condition references a field whose name was generated by the loop. For example:

```
dataKey: '$likes-{{#for-each-fruit.value}}'
```

At runtime this resolves to `$likes-apple` (or `$likes-banana`, etc.) based on the current iteration context. The resolution happens transparently when the condition is evaluated.

---

## Known limitations

### Phase 2 empty check vs full type validation for conditional fields

For fields inside conditionals, Phase 2 only checks whether the value is empty (undefined, null, empty string, or empty array). It does not re-run the full type and constraint rules from Phase 1 (e.g. regex pattern, min length). A conditional text-input with a regex constraint will have that constraint enforced only if the field is also present in the static schema — which it is not when `required: false`, or in complex nested layouts where the key cannot be statically determined. In practice this means format errors on conditional fields are not surfaced at submit time unless Phase 1 also covers them.

## Tests

`lib/specs/validation.test.ts` covers the full module. The test groups map directly to the concepts above:

- Each component type (text-input, slider, single-checkbox, etc.) — required/optional, constraints, custom error messages
- Group — nested fields included in schema
- Conditional — optional in static schema, enforced in Phase 2 when condition is met
- Static for-each — N fields generated, `{{#id.value}}` and `{{#id.index}}` resolved
- Dynamic for-each — fields validated at submit time, source absent/empty, per-iteration independence, error paths
- Dynamic for-each inside a group — found and validated regardless of nesting
- Dynamic for-each inside a conditional — only validated when the enclosing branch is active
- Conditional inside a static for-each — condition resolved per iteration using the loop's current value
- Nested conditionals — inner condition only evaluated when the outer branch is entered
- Nested dynamic for-each inside a static for-each — outer loop's value available when resolving inner template keys
